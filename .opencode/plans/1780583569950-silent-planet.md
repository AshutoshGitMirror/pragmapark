# Plan: Replicate Plan Mode's Reminder Mechanism for Autonomous Mode

## Goal
Make Autonomous mode's `<system-reminder>` injection work **identically** to Plan mode's — injected as a `synthetic: true` text part into the last user message on every turn, wrapped in `<system-reminder>` tags, visible in the conversation, every single turn.

## Background: How Plan Mode Works (built-in)

The built-in `SessionReminders.apply()` in `reminders.ts` runs at `prompt.ts:1351` — directly in the conversation loop, before the assistant message is created. It:

1. Finds the **last user message** (`messages.findLast(msg => msg.info.role === "user")`)
2. Pushes a **`synthetic: true` text part** with `<system-reminder>...</system-reminder>` content
3. The part flows through `toModelMessagesEffect` at line 1449 (which does **not** filter `synthetic`)
4. The model receives the `<system-reminder>` as plain text in the user message
5. At line 1429, multi-step wrapping **skips** `synthetic` parts to avoid double-wrapping

Key detail: there is NO built-in Autonomous mode support. Only `"plan"` and `"build"` agents are handled.

## Our Plugin's Approach

**File**: `~/.config/opencode/plugins/autonomous-reminder.ts`

The plugin should use **`experimental.chat.messages.transform`** (fires at `prompt.ts:1443`, after `SessionReminders.apply()` and multi-step wrapping, but before `toModelMessagesEffect`) to inject a synthetic `<system-reminder>` part into the last user message — **exactly** like Plan mode does.

### Implementation

1. **Event handler** (`session.next.agent.switched`): Track `currentAgent`, set `justSwitchedToAutonomous` flag
2. **`messages.transform` handler**:
   - If not Autonomous, return early
   - Find last user message in `input.messages`
   - Check if any existing part already contains `AUTONOMOUS_MARKER` (dedup within a turn / across tool-call iterations)
   - If already present, skip (prevents accumulation during tool-call loops)
   - Push a new part with:
     ```ts
     {
       partID: "prt" + Date.now(),  // or a monotonic ID
       messageID: userMessage.messageID,
       sessionID: userMessage.sessionID,
       type: "text",
       text: `<system-reminder>\nAUTONOMOUS STEWARDSHIP MODE\n...\n</system-reminder>`,
       synthetic: true,
     }
     ```
   - On switch: include switch announcement text before the directive
   - On every subsequent turn: just the directive wrapped in `<system-reminder>`

3. **Message part structure**: Must match the `Part` type from `message-v2.ts` — specifically `partID` must be a `PartID` type (monotonic ascending, prefix `"prt"`). We can use `PartID.ascending()` or just use a timestamp-based ID.

### Dedup Strategy

Within a single turn (multiple tool-call iterations), `messages.transform` fires each time. To avoid pushing the part repeatedly:
- Search `userMessage.parts` for any part whose text includes `AUTONOMOUS_MARKER`
- If found, skip pushing

Across turns (different user messages), each new user message gets a fresh injection — same as Plan mode.

### Redundancy: Keep `system.transform`

Optionally keep the `system.transform` hook as well, so the directive appears BOTH as:
- A `{role: "system"}` message (high authority, invisible to user)
- A `<system-reminder>` text part in the user message (visible, Plan-mode-style)

The user specifically asked for the visible `<system-reminder>` mechanism. The `system.transform` can stay as secondary reinforcement.

## Why This Works

The model is trained to treat `<system-reminder>` XML tags as authoritative directives. Plan mode uses this exact mechanism. The injection on **every single turn** reinforces it. The user confirmed this approach works — they're in Plan mode right now and can see the `<system-reminder>` text in the conversation.
